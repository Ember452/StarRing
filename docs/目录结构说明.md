# StarRing Phase 1 — 目录结构说明

## 设计理念

采用 **FastAPI 行业标准工程结构**，遵循：
- **app/** 统一包根：所有业务代码在 `app/` 下，与配置文件分离
- **api/config/core/models/schemas/db** 职责分离：路由 / 配置 / 异常 / ORM / 序列化 / 数据库各自独立
- **API 版本化**：`api/v1/` 预留未来版本升级空间
- **分层架构自顶向下**：接入 → 编排 → Agent → 基础（RAG/工具/模型/DB） → 持久（storage/data）
- **scripts/ tests/** 独立：运维脚本与测试套件在 app 外层，不污染业务代码

### 与旧结构的差异（为什么改）

| 旧结构问题 | 新结构改进 |
|-----------|-----------|
| 业务代码散落在根级目录 | 统一收入 `app/` 包，IDE 自动识别为 Python 包 |
| `models/` 混放 ORM + Pydantic | 拆分为 `app/models/` (ORM) + `app/schemas/` (Pydantic) |
| 无 API 路由层 | 新增 `app/api/v1/`，含版本化路由 + 聚合 router |
| 无核心配置层 | 新增 `app/config/`（配置中心）+ `app/core/`（异常/中间件/安全），Phase 2+ 配置拆分做准备 |
| 无测试目录 | 新增 `tests/`，conftest.py fixture 体系 |
| `tools/` 命名模糊 | 更名为 `app/utils/`，符合行业惯例 |
| DB 文件散落根目录 | 统一收入 `data/` 目录 |
| Chroma 向量数据在根目录 | 收入 `data/chroma/`，与 DB 同级 |
| Alembic 迁移无处放置 | 新增 `app/db/migrations/` 预留 |

## 分层原则

```
┌─────────────────────────────────────────────┐
│  接入层  app/api/        路由 / 请求验证     │
├─────────────────────────────────────────────┤
│  编排层  app/orchestration/  LangGraph 调度  │
├─────────────────────────────────────────────┤
│  Agent层 app/agents/      业务 Agent 实现    │
├──────────┬──────────┬───────────────────────┤
│  基础层  │          │                       │
│  rag/    │ models/  │ schemas/  utils/  db/ │
│  (向量)  │ (ORM)    │ (序列化)  (工具)  (库)│
├──────────┴──────────┴───────────────────────┤
│  横向   config/  配置    |  core/  异常/安全│
├─────────────────────────────────────────────┤
│  持久层  storage/ + data/                    │
└─────────────────────────────────────────────┘
```

### 依赖方向（核心原则）

1. **api/** → 调用 `orchestration/`，不直接访问 `agents/`
2. **orchestration/** → 调度 `agents/`，不关心 Agent 内部实现
3. **agents/** → 调用 `rag/` + `utils/`，不直接操作 `db/`
4. **models/ ↔ schemas/** → 互不依赖，由 `api/` 负责 ORM ↔ Pydantic 转换
5. **config/** → 被所有层引用，不引用任何层（与 core/ 并列，配置和异常分离）
6. **db/session.py** → 由 `api/` 通过 FastAPI Depends 注入，Agent 不持有连接

## 各模块角色与职责

### 接入层 `app/api/v1/`

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/tasks` | POST | 提交自然语言需求 → 创建 Task → 触发编排流水线 |
| `/api/v1/tasks/{id}` | GET | 查询任务状态 + 输出文件列表 |
| `/api/v1/health` | GET | 服务存活 + DB/Chroma 连通性检查 |

### 编排层 `app/orchestration/`

| 要素 | 说明 |
|------|------|
| 技术 | LangGraph StateGraph |
| 节点 | requirement → coding → testing → documentation（串行 4 节点） |
| 状态 | TypedDict `TaskState` 在节点间传递 |
| Phase 1 限制 | 无 Checkpointer、无中断恢复、无并行 |

### Agent 层 `app/agents/`

| Agent | 输入 | 输出 |
|-------|------|------|
| `requirement.py` | 自然语言需求 | PRD + 表结构 + API 清单 |
| `coding.py` | PRD + RAG 上下文 | Java Controller/Service/Mapper/Entity |
| `testing.py` | API 清单 + 生成代码 | JUnit + MockMvc 测试代码 |
| `documentation.py` | 生成代码 | OpenAPI YAML + DB 设计 Markdown |

每个 Agent 是纯函数：`(TaskState) -> TaskState`，不持有状态。

### RAG 层 `app/rag/`

| 模块 | 职责 |
|------|------|
| `client.py` | Chroma 客户端初始化、Collection 管理 |
| `indexer.py` | tree-sitter AST 解析 → embedding 向量化 → Chroma 入库 |
| `retriever.py` | 语义查询 → Top-K 相似代码片段 |

### 数据模型层 `app/models/` + `app/schemas/`

| 层 | 类型 | 用途 |
|----|------|------|
| `models/` | SQLAlchemy ORM | 数据库表映射（task, task_node） |
| `schemas/` | Pydantic BaseModel | API 请求/响应 JSON Schema |

分离原因：ORM 模型含数据库字段约束，API Schema 含验证和文档。混合会导致单一职责破坏。

## 依赖清单

### 已安装

| 包 | 用途 |
|---|------|
| `fastapi` | Web 框架 |
| `uvicorn` | ASGI 服务器 |

### 待安装（`uv add`）

| 包 | 用途 |
|---|------|
| `langgraph` | Agent 流水线编排 |
| `chromadb` | 向量数据库 |
| `openai` | SiliconFlow API 调用（兼容 OpenAI SDK） |
| `sqlalchemy` | ORM |
| `aiosqlite` | SQLite 异步驱动 |
| `python-dotenv` | .env 环境变量加载 |
| `pydantic-settings` | 类型安全配置管理 |
| `tree-sitter` | 代码 AST 解析核心 |
| `tree-sitter-java` | Java 语法解析 |
| `javalang` | Java 代码解析备选 |

```bash
cd StarRing
uv add langgraph chromadb openai sqlalchemy aiosqlite python-dotenv pydantic-settings tree-sitter tree-sitter-java javalang
```

## 启动流程

```bash
# 1. 安装依赖
cd StarRing && uv sync

# 2. 配置 API Key
# 编辑 .env, 填入 SILICONFLOW_API_KEY

# 3. 初始化数据库
uv run python scripts/init_db.py

# 4. 启动服务
uv run uvicorn main:app --reload --host 0.0.0.0 --port 8000

# 5. 访问
# Swagger UI:  http://localhost:8000/docs
# ReDoc:       http://localhost:8000/redoc
# Health:      http://localhost:8000/api/v1/health
# REST Client: 使用 test_main.http 发送请求

# 6. 运行测试
uv run pytest tests/ -v
```
